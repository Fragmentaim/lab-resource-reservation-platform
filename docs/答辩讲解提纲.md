# 答辩讲解提纲

## 1. 项目背景

我们的项目是实验室资源预约系统。它解决的是实验室资源使用过程中“资源不好查、时间不好约、预约状态不好管理”的问题。系统分为普通用户端和管理员端，普通用户负责预约资源，管理员负责维护资源、时段和用户。

## 2. 主要业务流程

可以按下面顺序讲：

1. 用户登录系统。
2. 用户浏览实验室资源列表。
3. 用户进入资源详情，选择一个开放的预约时段。
4. 系统校验时段状态和剩余名额。
5. 校验通过后生成预约记录，并扣减时段剩余名额。
6. 用户可以在“我的预约”中查看预约记录。
7. 如果用户取消预约，系统修改预约状态，并释放对应时段名额。

## 3. 数据库设计重点

重点讲 6 张核心表：

- `sys_user`：用户表
- `resource`：实验室资源表
- `resource_slot`：预约时段表
- `reservation`：预约记录表
- `user_notification`：用户通知表
- `sys_dict_type` / `sys_dict_data`：字典表

讲关系时重点说：

- 用户和预约是一对多。
- 资源和时段是一对多。
- 用户、资源、时段通过预约记录连接起来。
- 字典表用于维护资源类型、资源状态、时段状态等基础数据。

## 4. 可以展示的页面

建议现场演示这些页面：

1. 登录页
2. 资源列表页
3. 资源详情页
4. 我的预约页
5. 管理端资源管理页
6. 管理端时段管理页
7. 管理端预约管理页
8. 字典管理页

## 5. 不建议重点展开的内容

下面这些能力代码中存在，但数据库课程里不必深入讲：

- Redis 缓存和限流
- RocketMQ 异步预约
- 审计日志 Outbox
- 高并发压测
- 消息消费幂等

如果老师问到，可以简单回答：这些属于工程扩展能力，本次课程汇报重点放在数据库表设计和业务数据流转上。

## 6. 结束总结

可以这样收尾：

本系统围绕实验室资源预约场景，设计了用户、资源、时段、预约、通知和字典等核心实体。通过预约记录表连接用户、资源和时段，实现了预约创建、预约查询、取消预约和后台管理等功能。数据库设计中使用主键、唯一约束、状态字段和剩余名额字段保证业务数据的一致性，能够满足课程项目对数据库建模和业务实现的要求。
